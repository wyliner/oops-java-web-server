# 开发平台数据库设计文档v1

公司内部开发者以及OME开发者都可以访问开发平台，开发平台提供了一个统一的界面来管理和使用各种开发工具和资源。为了满足开发平台的需求，我们需要设计一个合理的数据库结构来存储和管理相关信息。开发平台的数据库设计需要满足以下需求：

## 前言

- 此平台前后端都需要支持多语言（至少中文和英文），因此所有面向用户展示的名称、描述等文本信息均使用 i18n Key
  存储，前端根据用户语言环境进行翻译展示。
- 所有时间字段统一使用 bigint 存储，单位为毫秒，避免时区问题和数据库兼容性问题。
- 用户、角色、组织等核心表采用逻辑删除（is_deleted 标记），保留历史记录，便于审计和数据恢复。
- 权限只能授予角色，用户通过角色间接获得权限，不允许直接赋权给用户，确保权限管理的清晰和可维护性。
- 所有关键写操作均记录审计日志，保留操作前后数据快照，满足合规审计需求。
- 权限、配置等高频读取数据配合 Redis 缓存，降低数据库压力，提高系统性能。

---

## 一、用户模块（User）

开发平台同时服务于公司内部开发者和 OEM 开发者，因此用户体系需要区分不同类型的开发者身份，并提供统一的账号管理能力。

**功能需求：**

- 用户可通过邮箱进行注册和登录，注册时需要验证码验证, 发送邮件功能需要独立提供接口方便接入第三方邮件服务
- 支持用户名、邮箱、手机号多种登录方式，目前只允许使用邮箱登录，后续可根据需求增加手机号登录
- 支持第三方账号（如 GitHub、Google）授权登录，暂时不开放注册，后续根据需求评估是否开放注册功能
- 支持二次认证（2FA），如邮箱验证码、短信验证码、TOTP 等
- 用户可自行开启或关闭二次认证，系统记录上次使用的认证方式以便下次默认展示
- 用户有启用/禁用状态，禁用状态下无法登录
- 用户可以属于多个组织，在不同组织内拥有不同角色
- 用户可在不同组织之间切换，访问对应组织的资源
- 超级管理员可管理所有组织、用户、角色和资源

## 二、组织模块（Organization）

平台以组织（即公司/团队）为核心单元进行资源隔离。内部团队和 OEM 客户均以组织形式接入平台，不同组织之间的数据和资源完全隔离。

**功能需求：**

- 一个组织只能有一个所有者（Owner），所有者拥有组织内一切管理权限
- 组织可以拥有多个管理员（Admin），管理员可管理组织内的用户、角色和资源
- 用户可申请加入组织，或由管理员直接邀请加入
- 组织有启用/禁用状态
- 超级管理员可跨组织分配资源，普通管理员只能管理自身组织内的资源
- 组织可设置基本信息，如名称、Logo、简介、联系方式等
- 支持组织层级（可选：父子组织关系）
- 父组织的管理员自动拥有子组织的管理权限，但子组织管理员只能管理自己组织内的资源
- 组织内用户的权限由其在该组织内的角色决定，不同组织内同一用户的权限可以不同
- 组织的资源（如角色、权限、功能资源等）与其他组织完全隔离，无法跨组织访问

---

## 三、组织成员模块（Organization Member）

记录用户与组织的多对多关系，以及用户在该组织内的角色和加入方式。

**功能需求：**

- 一个用户可以属于多个组织，同一用户在不同组织内可拥有不同角色
- 记录用户加入组织的方式（申请/邀请）及审批状态
- 成员退出或被移除后保留历史记录

## 四、角色模块（Role）

角色是权限的集合，通过角色管理实现灵活的 RBAC（基于角色的访问控制）体系。

**功能需求：**

- 角色归属于特定组织，不同组织的角色相互隔离
- 系统内置若干预置角色（如超级管理员、组织管理员、普通成员、只读成员）
- 组织管理员可在组织内自定义创建角色
- 角色可以聚合多个其他角色（角色组合），聚合后的角色拥有所有子角色的权限合集
- 角色中的某个子角色可以随时被移除
- 角色有启用/禁用状态，禁用后该角色下的用户权限自动失效
- 角色可以设置描述信息，便于管理员理解该角色的用途

---

## 五、角色聚合关系表（Role Composition）

记录角色之间的包含关系，支持将多个角色组合为一个新角色。

**功能需求：**

- 一个角色可以包含（引用）多个子角色
- 需要防止循环引用（A 包含 B，B 不能再包含 A）
- 移除聚合关系时不影响子角色本身

**核心字段：**

- 父角色 ID
- 子角色 ID
- 创建时间（毫秒级 bigint）

---

## 六、用户角色关联模块（User Role）

记录用户在某个组织内被分配的角色，支持时效性角色分配。

**功能需求：**

- 一个用户在同一组织内可拥有多个角色
- 用户的最终权限为所有角色权限的合集
- 角色分配支持时效性（有效期），到期后自动失效
- 记录分配人信息，便于审计追溯

**核心字段：**

- 用户 ID
- 组织 ID
- 角色 ID
- 有效期开始时间、结束时间（毫秒级 bigint，为空表示永久）
- 分配人用户 ID
- 创建时间（毫秒级 bigint）

---

## 七、功能资源模块（Resource / Menu）

功能资源是平台内所有可被权限控制的操作单元，涵盖菜单、页面、按钮及页面内部元素。

**功能需求：**

- 资源类型包括：菜单（menu）、页面（page）、按钮（button）、页面内元素（element）
- 资源支持树形层级结构（父子关系），菜单可以嵌套
- 每个资源对应唯一一个权限，权限与资源一一对应
- 资源数据可用于生成前端路由配置，兼容 Vue / React / Angular 等框架
- 资源可以排序，控制菜单显示顺序
- 资源有启用/禁用状态
- 支持 i18n 国际化，资源名称使用 i18n key 存储

**核心字段：**

- 资源唯一 ID
- 父资源 ID（树形结构，顶层为空）
- 资源名称 i18n Key
- 资源类型（menu / page / button / element）
- 资源路径（前端路由路径或组件标识）
- 图标
- 排序值
- 状态（启用 / 禁用）
- 创建时间、更新时间（毫秒级 bigint）

---

## 八、权限模块（Permission）

权限定义了对资源的具体操作能力，权限与功能资源一一对应，并区分操作类型。

**功能需求：**

- 权限只能分配给角色，不能直接分配给用户
- 权限明确区分读（read）和写（write / create / update / delete）操作
- 权限可以有层级关系（父权限 / 子权限）
- 页面访问权限可细分到按钮操作级别
- 权限有启用/禁用状态

**核心字段：**

- 权限唯一 ID
- 关联资源 ID
- 父权限 ID（层级关系）
- 权限名称 i18n Key
- 权限标识码（如 `user:create`、`user:read`）
- 操作类型（read / write / create / update / delete / manage）
- 状态（启用 / 禁用）
- 创建时间、更新时间（毫秒级 bigint）

---

## 九、角色权限关联表（Role Permission）

记录角色与权限的多对多关系。

**功能需求：**

- 一个角色可以拥有多个权限
- 同一权限可以被多个角色引用
- 权限变更后，对应角色下的所有用户访问控制实时生效

**核心字段：**

- 角色 ID
- 权限 ID
- 创建时间（毫秒级 bigint）

---

## 十、系统配置模块（System Config）

系统配置表存储全局或各模块的动态配置项，支持运行时修改，无需重启服务。

**功能需求：**

- 以键值对方式存储配置项
- 配置项有描述信息，便于管理员理解用途
- 支持热更新，修改后通过缓存同步到服务（Redis）
- 仅超级管理员可操作系统配置

**核心字段：**

- 配置唯一 ID（自增）
- 配置键（config_key，全局唯一）
- 配置值（config_value）
- 配置描述
- 更新时间（毫秒级 bigint）

---

## 十一、枚举类型模块（Enum Type）

集中管理系统中各类枚举值，支持 i18n 国际化显示，便于前端动态渲染下拉选项等 UI 组件。

**功能需求：**

- 枚举值按分组管理，每个分组对应一种业务类型（如用户状态、组织类型等）
- 枚举项名称和描述使用 i18n Key，支持多语言展示
- 枚举项支持排序控制显示顺序
- 枚举项有启用/禁用状态
- 枚举值类型可以是 int、bool、string 等

**核心字段：**

- 枚举唯一 ID（自增）
- 枚举分组标识（enum_group，如 `user.status`）
- 枚举项代码名称 i18n Key（enum_code）
- 枚举项值（enum_value，如 `ENABLE`、`DISABLE`）
- 值类型（value_type：int / bool / string）
- 描述
- 排序值
- 是否启用

---

## 十二、登录日志模块（Login Log）

记录每次用户登录行为的详细信息，用于安全审计和异常检测。

**功能需求：**

- 记录每次登录的时间、IP 地址、设备信息、地理位置
- 记录使用的登录方式（邮箱/手机号/第三方）
- 记录使用的二次认证方式
- 记录登录结果（成功 / 失败及失败原因）
- 支持按用户、组织、时间范围等维度查询

**核心字段：**

- 日志唯一 ID（自增）
- 用户 ID
- 所属组织 ID
- 登录方式（email / phone / oauth_github 等）
- 二次认证方式（none / email_code / sms_code / totp 等）
- 登录结果（success / fail）
- 失败原因
- 客户端 IP
- User-Agent
- 登录时间（毫秒级 bigint）

---

## 十三、操作审计日志模块（Audit Log）

记录用户在平台内的关键操作行为，满足合规审计需求。

**功能需求：**

- 记录所有对资源的增删改操作
- 记录操作人、操作时间、操作对象、操作类型
- 记录操作前后的数据快照（JSON）
- 支持按操作人、资源类型、时间范围查询
- 仅管理员可查看审计日志

**核心字段：**

- 日志唯一 ID（自增）
- 操作人用户 ID
- 所属组织 ID
- 操作类型（create / update / delete / login / logout 等）
- 资源类型（user / role / permission / organization 等）
- 资源 ID
- 操作前数据（JSON 快照）
- 操作后数据（JSON 快照）
- 客户端 IP
- 操作时间（毫秒级 bigint）

---

## 十四、通知模块（Notification）

平台内各类事件（如申请加入组织、角色变更、系统公告等）会触发站内通知。

**功能需求：**

- 支持站内信通知（In-App Notification）
- 通知类型包括：系统公告、申请审批、角色变更、密码修改提醒等
- 通知有已读/未读状态
- 支持批量标记已读
- 重要通知可同时触发邮件或短信推送

**核心字段：**

- 通知唯一 ID（UUID）
- 接收用户 ID
- 发送方（system / user_id）
- 通知类型
- 通知标题 i18n Key
- 通知内容
- 关联资源类型及 ID（如关联某个审批请求）
- 是否已读
- 已读时间（毫秒级 bigint）
- 创建时间（毫秒级 bigint）

---

## 设计原则汇总

| 原则    | 说明                                                 |
|-------|----------------------------------------------------|
| 数据隔离  | 不同组织之间的数据和资源完全隔离，通过 organization_id 进行租户隔离         |
| 时间统一  | 所有时间字段使用 bigint 存储，单位为毫秒，避免时区问题                    |
| 权限最小化 | 权限只能授予角色，用户通过角色间接获得权限，不允许直接赋权给用户                   |
| 国际化支持 | 所有面向用户展示的名称、描述均使用 i18n Key，至少支持中文（zh-CN）和英文（en-US） |
| 软删除   | 用户、角色、组织等核心表采用逻辑删除（is_deleted 标记），保留历史记录           |
| 审计追溯  | 所有关键写操作均记录审计日志，保留操作前后数据快照                          |
| 缓存友好  | 权限、配置等高频读取数据配合 Redis 缓存，降低数据库压力                    |
